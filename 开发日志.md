## 四、Bug修复记录

### 播放相关
- **进度条拖动后弹回**: 引入`isDragging`状态分离，拖动时显示拖动位置，释放后才触发seek
- **进度条跳动**: 使用`sliderValue`独立管理UI进度，非拖拽时同步真实进度
- **播放队列删除不刷新**: 删除后同步更新`_currentIndex`和`PlaybackState.currentIndex`
- **播放列表拖拽中断**: 使用稳定Key(`song.id`)和状态提升修复
- **播放列表key重复**: 使用`"${index}_${song.id}"`避免重复
- **播放队列页面问题**: 重构为全屏页面，只显示当前播放歌曲及之后的歌曲
- **蓝牙耳机控制**: 依赖 Media3 MediaSession 自动路由媒体按键，无需手动编写 Callback
- **通知栏播放器**: 移除自定义 NotificationHelper，依赖 MediaSessionService 自动生成系统媒体通知，添加 Android 13+ POST_NOTIFICATIONS 权限申请

### 歌词相关
- **歌词无法显示**: 支持多种扩展名(.lrc/.LRC)和编码(UTF-8/GBK)
- **内嵌歌词读取**: 使用 MediaMetadataRetriever 提取内嵌歌词 (API 29+)
- **歌词页面重写**: 全屏歌词页面，逐行高亮、点击跳转、自动滚动同步
- **下滑关闭手势**: 下滑关闭手势仅在列表无法向下滚动时触发
- **无效歌词过滤**: 过滤纯数字等无效的内嵌歌词内容
- **双语歌词支持**: 合并时间戳相近的歌词行，中英双语分行显示
- **居中显示**: 当前播放歌词居中显示在屏幕中央偏上两行
- **长歌词自动缩放**: 根据字符数动态调整字体大小，确保一行显示完整
- **点击歌词播放**: 即使暂停状态点击歌词也会跳转并开始播放
- **LRC offset支持**: 解析 [offset:xxx] 标签修复歌词同步偏移问题
- **触摸事件穿透修复**: 为歌词页添加动画配置和活跃状态追踪，阻止返回后触摸事件穿透
- **长歌词显示不完整**: 移除 maxLines=1 限制，允许长歌词自动换行显示

### UI相关
- **顶部空白问题**: 移除`enableEdgeToEdge()`调用
- **字母索引拖拽手势失效**: 在`awaitPointerEventScope`内部访问`size.height`
- **首页字母索引失灵**: Box添加weight(1f)获取正确高度，用协程替代runBlocking避免闪退
- **FilterScreen滑动问题**: 改为单一LazyColumn统一滑动
- **专辑封面显示**: 使用`ContentUris.withAppendedId()`正确构建URI
- **Coil API修正**: 将AsyncImage替换为SubcomposeAsyncImage
- **标签管理页滚动条位置和拖拽**: 滚动条移动到右侧(CenterEnd)，添加pointerInput拖拽手势支持

### 功能相关
- **搜索框选择后未清除**: 选择搜索结果后清空搜索词并关闭键盘
- **标签弹窗排序**: 标签选择弹窗和标签管理页按歌曲数量从多到少排序
- **添加文件夹闪退**: 添加URI解析方法获取真实路径
- **最近播放不记录**: 播放时调用`incrementPlayCount()`更新记录
- **筛选无条件时无结果**: 框A/B/C都为空时显示所有歌曲
- **通知栏闪退**: 正确调用前台服务API，修复RemoteViews兼容问题
- **布尔筛选边缘情况**: 当只有C框有标签时，返回所有歌曲排除C标签歌曲
- **标签页歌曲数量**: TagItem显示每个标签包含的歌曲数量
- **播放页标签点击行为**: 点击胶囊标签打开标签管理弹窗而非删除标签
- **搜索框选择后未清除**: 选择搜索结果后清除搜索词并关闭键盘

### 任务九：多歌手歌曲分类问题

**问题描述**: 一首歌有多个歌手时，应将这首歌分别归类到多个歌手名下。

**涉及文件**: ArtistRepository.kt

**技术方案 (内存倒排索引)**:

> **核心原则**: 严禁使用 `LIKE '%歌手名%'` 模糊匹配，必须使用精确匹配。

**方案选择**: 在内存中构建 `Map<String(精确歌手名), List<Song>>` 倒排索引

**优势**:
- 数据实时响应歌曲列表变化
- 无需数据库迁移
- 精确匹配键，杜绝"李健"匹配到"李健宇"的 Bug

**歌手名解析规则**:
```kotlin
// 分隔符: 、 , / & ; feat. ft.
// 示例: "周杰伦、方文山" -> ["周杰伦", "方文山"]
// 示例: "Jay feat. Beyonce" -> ["Jay", "Beyonce"]
private fun parseArtistNames(artistString: String): List<String> {
    if (artistString.isNullOrBlank()) return listOf("Unknown Artist")

    return artistString
        .split(Regex("""\s*(?:、|,|/|&|;|feat\.|ft\.)\s*"""))
        .map { it.trim() }
        .filter { it.isNotBlank() }
        .takeIf { it.isNotEmpty() }
        ?: listOf("Unknown Artist")
}
```

### 任务十二：备份恢复跨设备不兼容

**问题描述**: 当前备份机制依赖 `MediaStore ID` 作为歌曲标识，在不同设备间导入备份时，标签和歌单的歌曲关联无法正确恢复。

**问题根因**:
- `SongTag` 存储的是 `songId`（MediaStore 音频文件 ID）
- `PlaylistSong` 同样存储 `songId`
- MediaStore ID 在不同设备、不同扫描时机可能不同
- 备份不包含歌曲本身，只包含关联关系

**涉及文件**: SettingsRepository.kt, BackupData

**技术方案**:

**方案一：保存 filePath 用于匹配（推荐）**
```kotlin
@Serializable
data class SongTagBackup(
    val songPath: String,  // 文件路径作为稳定标识
    val tagName: String,
    val addedAt: Long
)
```
- 导出时：将 `songId` 转换为 `filePath` 存储
- 导入时：通过 `filePath` 匹配当前设备歌曲

**方案二：保存 title + artist 用于模糊匹配**
```kotlin
@Serializable
data class SongTagBackup(
    val songTitle: String,
    val songArtist: String,
    val tagName: String,
    val addedAt: Long
)
```
- 适用于文件路径变化但歌曲信息不变的场景
- 可能存在同名歌曲误匹配问题

**方案三：混合方案（最健壮）**
- 优先用 `filePath` 精确匹配
- 匹配失败时用 `title + artist` 模糊匹配

- [x] 修改 `createBackupData()` 转换 songId 为稳定标识（filePath + title + artist）
- [x] 修改 `restoreBackupData()` 实现智能匹配逻辑（路径优先，标题+歌手备选）
